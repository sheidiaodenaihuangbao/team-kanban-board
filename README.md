# team-kanban-board
软件需求作业
1. 学习目标
   完成本次作业后，团队成员应能够：
   熟练使用 Git 进行个人代码版本管理（clone, add, commit, push, pull）。
   理解并实践 Git 分支策略，为不同功能创建独立分支。
   掌握 Fork 和 Pull Request (PR) 的标准协作工作流。
   学会如何进行有效的代码审查，并通过评论进行建设性沟通。
   学习如何处理简单的合并冲突。
   体验基于 Git 的团队协作流程，理解其在项目管理中的重要性。
2. 项目背景
   你们是一个新成立的小团队，需要开发一个简单的“团队任务看板”应用来管理自己的项目进度。这个应用需要能够让团队成员创建、移动和删除任务卡片。为了确保代码质量和协作效率，你们决定采用业界标准的 Git + GitHub/GitLab + Pull Request 工作流。
3. 团队组成与角色
   请自行组建 3-5 人的小组，并分配以下角色：
   项目负责人: 负责创建中央仓库、管理分支保护规则、审查并合并所有 Pull Request、协调团队进度。
   前端开发者 A: 负责看板的 UI 布局和任务卡片的展示功能。
   前端开发者 B: 负责实现“创建新任务”和“删除任务”的功能。
   前端开发者 C: 负责实现“拖拽任务卡片以更新状态”的功能。
   如果团队只有3人，项目负责人可以兼任一个开发角色。
4. 项目技术栈
   为了让大家专注于 Git 和 PR 流程，技术栈应保持简单：
   前端: HTML5, CSS3, Vanilla JavaScript (不使用任何框架，降低复杂度)。
   数据存储: 使用浏览器的 localStorage，无需后端和数据库。
   版本控制平台: GitHub 或 GitLab (团队自选)。
   版本控制工具: Git。
5. 核心工作流与 Git/PR 使用规范
   这是本次作业的核心评分部分，请严格遵守！
   第一步：初始化项目 (项目负责人)
   在 GitHub/GitLab 上创建一个新的公开仓库，命名为 team-kanban-board。
   初始化仓库时，添加一个 README.md 文件，简要介绍项目。
   在仓库设置中，为 main 分支启用分支保护规则，要求所有 PR 必须经过至少一人审查后才能合并。
   第二步：Fork 仓库 (所有开发者)
   每个开发者（包括项目负责人）都需要 Fork 这个中央仓库到自己的个人账户下。
   Fork 后，你拥有了一个独立的副本，可以自由提交代码。
   第三步：克隆与配置 (所有开发者)
   将自己 Fork 后的仓库克隆到本地：git clone https://github.com/YOUR_USERNAME/team-kanban-board.git
   进入项目目录，并添加原始仓库（你 Fork 的源）为 upstream，以便后续同步主分支的最新代码：cd team-kanban-board
   git remote add upstream https://github.com/PROJECT_OWNER_USERNAME/team-kanban-board.git
   确认远程仓库配置：git remote -v
# 应该能看到 origin (你的fork) 和 upstream (原始仓库)
第四步：功能开发分支 (所有开发者)
在开始开发任何功能前，必须先从 upstream 的 main 分支拉取最新代码，并创建一个新的功能分支。分支命名应清晰明了，例如 feature/ui-layout。git pull upstream main
git checkout -b feature/ui-layout
在该功能分支上进行编码。
第五步：提交代码 (所有开发者)
完成部分功能后，将修改提交到本地仓库。提交信息必须清晰、有意义。git add .
git commit -m "feat: add basic kanban board columns and header"
将你的功能分支推送到你自己的 Fork 仓库（即 origin）。git push origin feature/ui-layout
第六步：创建 Pull Request (PR) (所有开发者)
在 GitHub/GitLab 上，从你的功能分支向原始仓库的 main 分支创建一个 Pull Request。
PR 的标题和描述非常重要！
标题: 简明扼要地说明你做了什么。
描述:
详细描述本次变更的内容和目的。
如何测试或查看你的新功能。
附上截图。
如果有未解决的问题，可以列出。

第七步：代码审查 (所有团队成员)
除了 PR 作者外的所有团队成员都必须参与审查。
在 PR 的 "Files changed" 标签页下仔细查看代码变更。
对代码有疑问、建议或发现问题时，使用评论功能进行具体到行的讨论。评论应保持专业和建设性。
PR 作者根据审查意见修改代码，并 push 新的提交到同一个功能分支。PR 会自动更新。
第八步：合并代码 (项目负责人)
当 PR 获得足够多的 "Approve" 后，由项目负责人负责合并。
推荐使用 "Squash and merge"（变基并合并）方式。这能将你的多个提交压缩成一个整洁的提交，并保留到 main 分支的历史中，使主分支历史更加清晰。
合并后，立即删除已合并的功能分支。
第九步：保持本地仓库同步 (所有开发者)
在 main 分支有更新后，其他开发者需要将这些更新同步到本地，并合并到自己的功能分支，以避免未来的合并冲突。git checkout main
git pull upstream main
# 切换回你的功能分支
git checkout feature/your-feature
# 将 main 的最新代码合并到你的功能分支
git merge main
6. 作业提交内容
   请以团队为单位，提交一个压缩包，包含以下内容：
   项目最终仓库链接: 指向项目负责人创建的原始仓库 team-kanban-board。
   Pull Request 列表: 一个文档（如 Markdown 或 PDF），列出团队创建的所有 PR 的链接，并简要说明每个 PR 的贡献者和功能。
   团队协作报告 (1-2页):
   团队成员及角色分配。
   在协作过程中遇到了哪些挑战（例如：合并冲突、沟通问题）？你们是如何解决的？
   每个成员谈谈自己对 Git/PR 工作流的理解和收获。
   你们认为本次工作流有哪些可以改进的地方？
7. 评分标准
   Git 工作流规范性 (40%):
   是否正确使用 Fork 和 PR 模型。
   分支命名是否规范，是否在功能分支上开发。
   提交信息是否清晰。
   PR 描述是否详尽。
   代码审查与协作 (30%):
   是否每个成员都参与了代码审查。
   审查评论是否具有建设性。
   团队沟通是否顺畅。
   项目功能完整性 (20%):
   最终的看板应用是否实现了所有基本功能（创建、删除、移动任务）。
   代码质量是否良好。
   团队协作报告 (10%):
   报告内容是否真实、深刻，能体现出团队对整个流程的反思。
   祝你们协作愉快！ 这个作业不仅是技术的考验，更是团队沟通和协作能力的锻炼。遇到问题时，请首先尝试在团队内部讨论解决，这是学习过程的重要部分。
